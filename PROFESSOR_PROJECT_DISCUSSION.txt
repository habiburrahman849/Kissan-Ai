2. PROBLEM STATEMENT
--------------------------------------------------------------------------------

Agriculture employs a large share of Pakistan’s workforce, but many small and
medium farmers still lack timely, personalized advisory support.

Common challenges:

  • Extension officers are limited relative to the number of farmers.
  • Advice is often generic, not tied to a farmer’s district, crop, soil,
    irrigation method, or past treatments.
  • Farmers communicate in English, Urdu script, and Roman Urdu (Hinglish),
    while most digital tools are English-first.
  • Weather decisions (spray timing, irrigation) are hard without local
    guidance.
  • Generic chatbots forget context between sessions and may invent non-local
    recommendations.

Research / impact opportunity:

  Can a multilingual AI agent maintain season-long memory of a farm and
  ground answers in local agricultural documents, so advice improves over time?

3. PROJECT OVERVIEW — WHAT IS KISSAN AI?
--------------------------------------------------------------------------------

Kissan AI is a web-based intelligent agricultural assistant designed mainly
for Pakistani farmers.

Core idea:
  A “MemoryAgent” that remembers each farmer’s profile and crop history, then
  answers questions using:

    (A) Farmer-specific memory (who they are, what they grow, what failed)
    (B) Local scientific / government knowledge (PDF corpus + retrieval)
    (C) A large language model (Qwen via Alibaba DashScope API)
    (D) Weather context (OpenWeatherMap)

In simple terms:
  Chat with a farming assistant that speaks the farmer’s language, recalls
  previous conversations and farm details, and tries to base advice on
  Pakistan-relevant agricultural material.


Example interaction (conceptual):

  Farmer: “I sowed cotton FH-142 in Multan last week. When should I apply urea?”
  System: Stores crop, variety, location, sowing time in memory; retrieves
          fertilizer guidance; answers in the same language style as the user.

  Later (new chat session, same farmer):
  Farmer: “Yellow spots on leaves — what should I do?”
  System: Recalls cotton FH-142 + Multan + last fertilizer, and answers with
          that context instead of asking everything again.

That recall behavior is the academic and hackathon focus of the project.

4. OBJECTIVES
--------------------------------------------------------------------------------

Primary objectives:
  1. Build a working multilingual farming chat assistant (EN / Urdu / Roman Urdu).
  2. Implement persistent farmer memory across sessions (MemoryAgent track).
  3. Ground answers in a local PDF knowledge base (RAG — Retrieval Augmented
     Generation).
  4. Integrate Qwen (Alibaba Cloud / DashScope) as the core LLM.
  5. Provide a simple mobile-friendly web interface for demo and user testing.

Secondary objectives:
  • Weather-aware field advisories
  • Crop cycle tracking (sowing → observations → harvest)
  • Optional image-based crop problem support (planned; not fully complete)
  • Safety-aware responses (e.g., caution on pesticides, refer to local experts)

6.1 Frontend
------------
Static multi-page website:
  • index.html     — sign-in / registration style onboarding
  • chat.html      — main conversation interface
  • weather.html   — weather and field advisories
  • profile.html   — farmer profile editing
  • settings.html  — preferences
  • help.html      — help / FAQ
  • shared.js/css  — common UI, theme, API calls

Design style: mobile-first, chat-like interface (similar to messaging apps),
dark/light theme support.


6.2 Backend
-----------
Framework: FastAPI (Python)
Database: SQLite via SQLAlchemy

Main API areas:
  • /api/chat/*       — send message, chat history
  • /api/farmers/*    — farmer profile get/update
  • /api/crops/*      — start/harvest crop cycle
  • /api/documents/*  — upload agri documents
  • /api/weather/*    — current weather + advisories
  • /api/health       — health check


6.3 Memory layer (MemoryAgent core)
-----------------------------------
Two levels:

  1) Structured local memory (SQLite)
     - Farmer profile: name, district, village, land size, soil, irrigation
     - Crop cycles: crop name, variety, sowing date, active/archived
     - Farm events: fertilizer, symptoms, observations, harvest
     - Chat history: last messages used as conversation context

  2) Optional semantic cloud memory (Mem0 API)
     - Stores free-text conversation memories
     - Retrieves related past memories by query

Fact extraction from chat is currently rule/keyword based (crops, urea,
symptoms, harvest words). A stronger LLM-based extraction step is planned.


6.4 Knowledge layer (RAG)
-------------------------
Corpus: ~30 Pakistan-related agricultural PDFs, including examples such as:
  • Crop calendars (Kharif / Rabi)
  • Fertilizer and land utilization tables
  • Area/production statistics
  • Women in agriculture materials
  • Related government / census style reports

Pipeline (intended):
  PDF text extraction → chunking → multilingual embeddings
  (sentence-transformers) → FAISS vector index → top-k retrieval at query time

Important current issue (see Section 9):
  The retrieval path and index build are not fully reliable yet; this is a
  top priority fix.


6.5 Language model (Qwen)
-------------------------
  • Provider: Alibaba DashScope (OpenAI-compatible endpoint)
  • Default model: qwen-plus
  • Role: generate final multilingual farming advice using:
      - system prompt (role, language rules, memory guidance, safety)
      - farmer memory
      - retrieved document snippets
      - recent chat history

If API key is missing or the call fails, the system falls back to a mock
development answer (useful for offline coding, but not for final demo).


6.6 Weather service
-------------------
OpenWeatherMap integration:
  • Current temperature, humidity, wind, rain
  • Short forecast
  • Simple farming advisories (e.g., delay spray if wind high; disease watch
    if humidity high)

8. STRENGTHS OF THE PROJECT
--------------------------------------------------------------------------------

1) Strong real-world problem
   Connects AI systems research (memory agents, RAG, multilingual NLP) to a
   socially meaningful domain: farmer advisory support in Pakistan.

2) Good fit for MemoryAgent track
   The project is not only “ask a question, get an answer.” It is designed
   around long-term farm state: profile, season, problems, and history.

3) Multilingual by design
   Explicit handling of English, Urdu script, and Roman Urdu is important for
   real users and is academically interesting for language detection and
   response style control.

4) Hybrid memory architecture
   Combining structured database memory (crop cycles, events) with optional
   semantic memory (Mem0) is a sensible systems design for agents.

5) Local knowledge orientation
   Using Pakistan agricultural PDFs (calendars, fertilizer tables, etc.) is
   better than relying only on general LLM world knowledge.

6) End-to-end product prototype
   Unlike a notebook-only prototype, there is a full web application path:
   onboarding → chat → profile → weather. This supports user studies and demos.

7) Safety awareness in prompting
   System prompt encourages caution on pesticides and referral to local
   extension / experts — important for responsible AI discussion.

9. WEAKNESSES AND CURRENT LIMITATIONS
--------------------------------------------------------------------------------

These points are intentionally frank, so academic feedback can be precise.

9.1 Retrieval (RAG) reliability
--------------------------------
  • Vector index may not be built (`index.faiss` missing).
  • Path mismatch risk between where ingest saves the index and where the
    retriever loads it.
  • Document upload does not automatically re-embed documents.
  Consequence: answers may not truly use the PDF knowledge base yet.

9.2 Memory intelligence is still shallow
----------------------------------------
  • Current fact extraction is mostly keywords/regex, not full LLM extraction.
  • Limited crop/symptom vocabulary.
  • Cross-session personalization works only for facts the system successfully
    captured.
  Consequence: “MemoryAgent” claim needs stronger demonstration.

9.3 Qwen is under-utilized relative to platform capability
----------------------------------------------------------
  Currently used mainly as one-shot chat completion.
  Not yet fully used:
    • function / tool calling
    • multimodal vision (Qwen-VL) for leaf/crop images
    • streaming responses
    • structured JSON extraction for memory updates
  Consequence: technical depth for a Qwen-sponsored hackathon is still limited.

9.4 Some UI features overstate readiness
----------------------------------------
  • Image upload currently leads to a “feature coming soon” style response.
  • Voice UI is mostly visual (timer / bars), not real speech recognition.
  • Help text may claim disease identification “within seconds,” which is not
    fully true yet.
  Consequence: academic and ethical risk of overclaiming.

9.5 Multi-user / authentication maturity
----------------------------------------
  • Demo often uses a single default farmer (farmer_id = 1).
  • Full secure multi-tenant auth is not production-grade.
  Consequence: fine for prototype; not yet a deployable multi-user system.

9.6 Evaluation and scientific rigor
-----------------------------------
  • Limited automated tests.
  • No formal accuracy evaluation against extension-officer answers.
  • No measured retrieval quality (precision/recall of chunks).
  • No user study results yet (usability, trust, language preference).
  Consequence: strong engineering prototype, weaker research validation so far.

9.7 Production / deployment readiness
-------------------------------------
  • Local SQLite MVP
  • Dependency versions not tightly pinned
  • Error handling sometimes silently falls back to mock answers
  • Public deployment packaging still needed for external judges/users

10. ACADEMIC / RESEARCH ANGLES (FOR PROFESSOR DISCUSSION)
--------------------------------------------------------------------------------

Possible research framing options:

A) Memory Agents for longitudinal agricultural advisory
   How should structured farm state + conversational memory be combined so
   advice remains consistent across a crop season?

B) Multilingual HCI for low-literacy / mixed-script users
   How do English, Urdu, and Roman Urdu affect trust, comprehension, and
   task success in farming chat interfaces?

C) Grounded generation with local agricultural corpora
   Does RAG over national agri documents reduce hallucination and improve
   locality of recommendations compared with LLM-only answers?

D) Responsible AI for high-stakes advice
   How should an agri assistant express uncertainty, cite sources, and
   escalate to human experts?

E) Multimodal crop problem support (future)
   Can vision-language models assist preliminary symptom description while
   remaining safe and non-diagnostic in a medical/agronomic sense?

If this project may later become an FYP / thesis / paper, options A–C are the
most natural with the current codebase.

13. DEMO SCRIPT (FOR LAB / PROFESSOR REVIEW)
--------------------------------------------------------------------------------

Goal: prove multilingual response + memory + (ideally) local knowledge.

Step 1:
  User (English):
  “I farm 4 acres in Multan. I sowed cotton variety FH-142 last week.”
  Expected: system acknowledges and stores profile/crop facts.

Step 2:
  User (Roman Urdu):
  “bhai urea kab dalun?”
  Expected: answer in Roman Urdu; references cotton age/location if stored.

Step 3:
  Open a new chat session with the same farmer ID.
  User:
  “What crop am I growing?”
  Expected: recalls cotton FH-142 / Multan without being retold.
  This is the MemoryAgent checkpoint.

Step 4 (optional):
  Ask weather-sensitive question:
  “Should I spray tomorrow?”
  Expected: uses weather risk (rain/wind) if available.

Step 5 (optional, only if vision ready):
  Upload leaf photo → model describes symptoms + cautious guidance.

14. ETHICAL AND SAFETY CONSIDERATIONS
--------------------------------------------------------------------------------

  • Agricultural advice can affect livelihoods; the system should not claim
    certainty it does not have.
  • Pesticide recommendations must be careful (banned products, dosage,
    local regulations).
  • Prefer “consult local extension officer / agronomist” for serious cases.
  • Cite sources when using document retrieval.
  • Avoid overclaiming medical/diagnostic style certainty for crop disease.
  • Respect user data privacy (farm location, phone, chat history).
  • Be transparent in academic presentation about what is prototype vs validated.

18. HONEST OVERALL ASSESSMENT
--------------------------------------------------------------------------------

What we have:
  A promising end-to-end prototype of a multilingual farming MemoryAgent,
  with a clear social problem, a coherent architecture, Qwen integration
  started, and a usable web interface.

What we do not yet have:
  Fully reliable document retrieval, deep memory extraction, advanced Qwen
  tool/multimodal usage, rigorous evaluation, and production-grade multi-user
  deployment.

Bottom line for academic discussion:
  The project direction is strong and hackathon-relevant. The next phase should
  focus less on adding new pages and more on making the core agent truthful,
  memory-driven, and measurable.
